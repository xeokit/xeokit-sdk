XKTLoaderPlugin — Fallback Colours per IFC Type

Assigning meaningful colours to elements that have no colour defined in the IFC model

Toggle

Concept

Some IFC models contain elements with no colour information — for example, models exported without material or surface-style assignments.

This example applies fallback colours at runtime via entity.colorize, using a per-IFC-type colour map. Elements that already carry a meaningful IFC colour (detected by checking whether their mesh colour has chroma) are left untouched, so only neutral/achromatic elements receive a fallback.

Additionally, a name-based override map lets you recolour specific elements regardless of their existing colour — useful for objects like Revit RPC trees that export as IfcBuildingElementProxy with a non-neutral placeholder colour. Name overrides are always applied and take priority over the type map.

The DEFAULT key in the type map acts as a catch-all for any IFC type that has no explicit entry.

Colour palette

The palette follows the Wong (2011) colorblind-safe categorical set, organised by visual role so that the scheme stays readable across projects:

  • Structure / fabric — neutral, low-saturation greys (#BDBDBD → #6E6E6E)
  • Openings / circulation — noticeable tones: sky-blue windows, vermillion doors, pink stairs
  • MEP / equipment — saturated accents: teal pipes, amber equipment, yellow sensors
  • Vegetation — muted forest green, always applied by name match (Revit RPC trees are exported as IfcBuildingElementProxy; IfcGeographicElement in IFC 4.3 is the proper long-term type)
  • Unknown / proxy — light grey #C7C7C7
